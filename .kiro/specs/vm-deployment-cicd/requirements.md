# Requirements Document

## Introduction

This feature covers a unified GitHub Actions CI/CD pipeline that provisions an Azure VM via Terraform, then deploys the CLEO application stack to that VM using Docker Compose and NGINX reverse proxy — all in a single manually-triggered workflow.

The pipeline handles the full lifecycle:
- **Infrastructure**: Terraform provisions the Azure VM (network, security group, VM with cloud-init installing Docker and PostgreSQL)
- **Application**: Docker Compose deploys the CLEO stack (frontend, backend, observability) on the provisioned VM via `docker-compose.observability.yml`
- **Routing**: NGINX reverse proxy routes subdomain traffic to the correct services
- **Teardown**: Destroy action cleans up application services then destroys infrastructure

The stack includes:
- Frontend: Next.js (port 3000) → `cleo.andinolabs.ai`
- Backend: FastAPI (port 8000) → `api.andinolabs.ai`
- Grafana: (port 3002) → `grafana.andinolabs.ai`
- Full observability stack (OpenTelemetry, Tempo, Loki, Prometheus, Grafana) via `docker-compose.observability.yml`
- PostgreSQL running on the VM (provisioned by cloud-init)

## Glossary

- **Pipeline**: The unified GitHub Actions workflow that orchestrates the full deploy and destroy lifecycle.
- **Terraform**: The infrastructure-as-code tool used to provision the Azure VM and related resources from `terraform/env/dev`.
- **VM**: The Azure Linux virtual machine provisioned by Terraform that hosts all Docker services.
- **cloud-init**: The VM bootstrap script that installs Docker, Docker Compose plugin, and PostgreSQL on first boot.
- **NGINX**: The reverse proxy installed on the VM that routes HTTP traffic by domain name to the appropriate Docker service.
- **Docker_Compose**: The container orchestration tool used to manage CLEO services on the VM.
- **Deploy_Action**: The pipeline action that provisions infrastructure and starts all services on the VM.
- **Destroy_Action**: The pipeline action that stops all services, removes configuration, and destroys infrastructure.
- **Deploy_Target**: The scope of a deploy operation — `all` deploys every service, while `backend`, `frontend`, or `observability` rebuilds and restarts only the specified service(s).
- **Backend_Env**: A JSON object containing runtime environment variables for the FastAPI backend service.
- **Frontend_Env**: A JSON object containing runtime environment variables for the Next.js frontend service.
- **Domain_Config**: The set of three domain names provided at pipeline runtime for frontend, backend, and Grafana.
- **cleo-network**: The Docker bridge network shared by all CLEO services.
- **Observability_Stack**: The set of services defined in `docker-compose.observability.yml` (OpenTelemetry Collector, Tempo, Loki, Prometheus, Grafana).
- **Terraform_Backend**: The Azure Storage account (`vanguardtfstate`, container `tfstate`, key `vanguard.terraform.tfstate`) used to store Terraform state.
- **SSH_Key**: The RSA private key generated by Terraform's TLS provider and output as the sensitive `ssh_private_key` output, used to SSH into the VM as `azureuser`.
- **GH_PAT**: A GitHub Personal Access Token with `repo` scope (which includes `secrets:write`), stored as a GitHub Actions secret. Used by the Pipeline to call the GitHub REST API and automatically save the Terraform-generated SSH private key as the `VM_SSH_PRIVATE_KEY` secret after the first deploy, so it is available for all future re-deploys and destroy operations without manual intervention.

### GitHub Actions Secrets Reference

| Secret | Description |
|---|---|
| `AZURE_CREDENTIALS` | Azure service principal credentials used for `azure/login` and Terraform operations. |
| `POSTGRES_PASSWORD` | PostgreSQL password passed to Terraform as `TF_VAR_postgres_password`. |
| `VM_SSH_PRIVATE_KEY` | Auto-saved after first deploy via `GH_PAT`; used for SSH access on re-deploys and destroy. |
| `GH_PAT` | Personal Access Token for writing the `VM_SSH_PRIVATE_KEY` secret via the GitHub REST API. Must be created manually once before the first pipeline run. |

## Requirements

### Requirement 1: Pipeline Trigger and Inputs

**User Story:** As a DevOps engineer, I want to manually trigger the unified CI/CD pipeline with configurable inputs, so that I can deploy or destroy the full CLEO stack on demand without modifying code.

#### Acceptance Criteria

1. THE Pipeline SHALL support manual triggering via `workflow_dispatch`.
2. WHEN the Pipeline is triggered, THE Pipeline SHALL accept an `action` input with allowed values `deploy` and `destroy`.
3. WHEN the Pipeline is triggered, THE Pipeline SHALL accept a `backend_env` input as a JSON string containing backend environment variables.
4. WHEN the Pipeline is triggered, THE Pipeline SHALL accept a `frontend_env` input as a JSON string containing frontend environment variables.
5. WHEN the Pipeline is triggered, THE Pipeline SHALL accept a `frontend_domain` input with default value `cleo.andinolabs.ai`.
6. WHEN the Pipeline is triggered, THE Pipeline SHALL accept a `backend_domain` input with default value `api.andinolabs.ai`.
7. WHEN the Pipeline is triggered, THE Pipeline SHALL accept a `grafana_domain` input with default value `grafana.andinolabs.ai`.
8. WHEN the Pipeline is triggered, THE Pipeline SHALL accept a `backend_version` input with default value `latest` and description "Backend service version to deploy (e.g. v1.2.0, latest)".
9. WHEN the Pipeline is triggered, THE Pipeline SHALL accept a `frontend_version` input with default value `latest` and description "Frontend service version to deploy (e.g. v1.3.0, latest)".
10. WHEN the Pipeline is triggered, THE Pipeline SHALL accept a `deploy_target` input with allowed values `all`, `backend`, `frontend` and default value `all`.
11. IF the `action` input is `deploy` and `backend_env` or `frontend_env` is not valid JSON, THEN THE Pipeline SHALL fail with a descriptive parse error before executing any remote commands.
12. IF the `deploy_target` is not `all` and the VM does not yet exist, THEN THE Pipeline SHALL fail with a descriptive error indicating that the first deploy must use `deploy_target=all`.

### Requirement 2: Azure Authentication and Terraform Initialization

**User Story:** As a DevOps engineer, I want the pipeline to authenticate with Azure and initialize Terraform before provisioning, so that infrastructure changes are applied against the correct Azure subscription and remote state.

#### Acceptance Criteria

1. WHEN the Deploy_Action or Destroy_Action begins, THE Pipeline SHALL authenticate with Azure using the `AZURE_CREDENTIALS` GitHub Actions secret via the `azure/login` action.
2. WHEN Azure authentication succeeds, THE Pipeline SHALL initialize Terraform in the `terraform/env/dev` working directory using the Azure Storage backend (`vanguardtfstate` storage account, `tfstate` container, `vanguard.terraform.tfstate` key).
3. IF Azure authentication fails, THEN THE Pipeline SHALL fail with a descriptive error before executing any Terraform commands.
4. IF Terraform initialization fails, THEN THE Pipeline SHALL fail with a descriptive error before executing any Terraform plan or apply commands.

---

### Requirement 3: Deploy Action — Infrastructure Provisioning

**User Story:** As a DevOps engineer, I want the pipeline to provision the Azure VM via Terraform only when it does not yet exist, so that infrastructure is created safely on first deploy and skipped on re-deploys.

#### Acceptance Criteria

1. WHEN the `action` input is `deploy`, THE Pipeline SHALL run `terraform init` to access the remote state backend before any provisioning decisions are made.
2. WHEN Terraform init completes, THE Pipeline SHALL run `terraform output -raw vm_public_ip` to determine whether the VM already exists in state.
3. IF `terraform output -raw vm_public_ip` returns a valid IP address, THEN THE Pipeline SHALL skip the Terraform plan, Trivy scan, and apply steps and use the existing IP for subsequent deployment steps.
4. IF `terraform output -raw vm_public_ip` returns no output or an error (VM does not exist), THEN THE Pipeline SHALL run `terraform plan -out=tfplan.binary` with `TF_VAR_postgres_password` set from the `POSTGRES_PASSWORD` GitHub Actions secret.
5. WHEN the Terraform plan succeeds on a first-time provision, THE Pipeline SHALL convert the plan to JSON and run a Trivy security scan against it for HIGH and CRITICAL severity findings before applying.
6. WHEN the Trivy scan completes on a first-time provision, THE Pipeline SHALL run `terraform apply -auto-approve tfplan.binary` to provision the VM.
7. WHEN Terraform apply completes on a first-time provision, THE Pipeline SHALL capture the `vm_public_ip` output as a step output for use in subsequent steps.
8. WHEN Terraform apply completes on a first-time provision, THE Pipeline SHALL capture the `ssh_private_key` sensitive output and store it as the `VM_SSH_PRIVATE_KEY` GitHub Actions secret for use in future destroy and re-deploy operations.
9. IF the VM already exists (re-deploy), THE Pipeline SHALL use the `VM_SSH_PRIVATE_KEY` GitHub Actions secret directly for SSH access without capturing a new key from Terraform outputs.
10. IF Terraform apply fails, THEN THE Pipeline SHALL fail with a descriptive error and skip all subsequent deployment steps.

### Requirement 4: Deploy Action — cloud-init Readiness Wait

**User Story:** As a DevOps engineer, I want the pipeline to wait for cloud-init to finish before attempting SSH-based deployment on a newly provisioned VM, so that Docker and PostgreSQL are fully installed before the application deployment begins.

#### Acceptance Criteria

1. WHEN the VM was newly created (Terraform apply ran), THE Pipeline SHALL wait for the VM's cloud-init process to finish before proceeding to SSH-based steps.
2. WHILE waiting for cloud-init on a newly created VM, THE Pipeline SHALL poll the VM via SSH at 30-second intervals, checking for the presence of `/var/log/cloud-init-complete.log`, until the file exists or a timeout of 10 minutes is reached.
3. IF the cloud-init readiness timeout is reached without the completion log being present, THEN THE Pipeline SHALL fail with a descriptive timeout error.
4. WHEN cloud-init readiness is confirmed on a newly created VM, THE Pipeline SHALL verify that the `docker` command is available on the VM before proceeding.
5. IF the VM already existed (re-deploy, Terraform apply was skipped), THE Pipeline SHALL skip the cloud-init wait entirely and proceed directly to SSH-based deployment steps.

---

### Requirement 5: Deploy Action — Secure SSH Connectivity

**User Story:** As a DevOps engineer, I want the pipeline to connect to the VM securely using the Terraform-generated SSH key, so that no credentials are exposed in repository code or pipeline logs.

#### Acceptance Criteria

1. WHEN establishing SSH connections during deploy, THE Pipeline SHALL use the `ssh_private_key` captured from Terraform outputs, written to a temporary file on the runner with permissions `600`.
2. THE Pipeline SHALL connect to the VM as the `azureuser` admin user at the IP address from the `vm_public_ip` Terraform output.
3. THE Pipeline SHALL use `StrictHostKeyChecking=no` for the initial connection to the newly provisioned VM.
4. WHEN the deploy job completes (success or failure), THE Pipeline SHALL remove the temporary SSH key file from the runner.
5. IF an SSH connection to the VM fails, THEN THE Pipeline SHALL fail with a descriptive error and exit without executing further remote commands.

---

### Requirement 6: Deploy Action — Repository Sync to VM

**User Story:** As a DevOps engineer, I want the pipeline to sync the repository to the VM, so that the latest Docker Compose files and application code are available for the build.

#### Acceptance Criteria

1. WHEN SSH connectivity is confirmed, THE Pipeline SHALL clone or sync the repository to a designated project directory on the VM (e.g., `~/cleo`).
2. IF the project directory already exists on the VM, THE Pipeline SHALL update it to the current commit rather than re-cloning.
3. THE Pipeline SHALL ensure the `docker-compose.observability.yml` file is present in the project directory before proceeding.

---

### Requirement 7: Deploy Action — Environment Variable Injection

**User Story:** As a DevOps engineer, I want backend and frontend environment variables passed at runtime to always be written to the VM on every deploy, so that the VM env state is always driven by the pipeline and never drifts from stale files.

#### Acceptance Criteria

1. WHEN the `action` input is `deploy`, THE Pipeline SHALL parse the `backend_env` JSON string and write each key-value pair as `KEY=VALUE` lines to `backend/.env` within the project directory on the VM, overwriting any existing file.
2. WHEN the `action` input is `deploy`, THE Pipeline SHALL parse the `frontend_env` JSON string and write each key-value pair as `KEY=VALUE` lines to `frontend/.env.local` within the project directory on the VM, overwriting any existing file.
3. THE Pipeline SHALL write env files on every deploy regardless of `deploy_target`, so that env vars are always up to date before any service is rebuilt.
4. THE Pipeline SHALL set file permissions on the written env files to `600` to restrict read access.

---

### Requirement 8: Deploy Action — NGINX Installation and Configuration

**User Story:** As a DevOps engineer, I want NGINX to be installed and configured automatically with the provided domain names, so that each subdomain routes to the correct service without manual server configuration.

#### Acceptance Criteria

1. WHEN the `action` input is `deploy`, THE Pipeline SHALL check whether NGINX is installed on the VM and install it via `apt-get` if it is not present.
2. WHEN the `action` input is `deploy` AND `deploy_target` is `all`, THE Pipeline SHALL generate and apply NGINX server blocks for all three domains.
3. WHEN `deploy_target` is `backend`, `frontend`, or `observability`, THE Pipeline SHALL skip NGINX reconfiguration as domains do not change on partial updates.
4. WHEN the `action` input is `deploy` AND `deploy_target` is `all`, THE Pipeline SHALL generate an NGINX server block for the `frontend_domain` that proxies requests to `localhost:3000`.
5. WHEN the `action` input is `deploy` AND `deploy_target` is `all`, THE Pipeline SHALL generate an NGINX server block for the `backend_domain` that proxies requests to `localhost:8000`.
6. WHEN the `action` input is `deploy` AND `deploy_target` is `all`, THE Pipeline SHALL generate an NGINX server block for the `grafana_domain` that proxies requests to `localhost:3002`.
5. THE NGINX configuration SHALL set `proxy_set_header Host`, `X-Real-IP`, and `X-Forwarded-For` headers for all proxied locations.
6. THE NGINX configuration for the `backend_domain` SHALL include `proxy_http_version 1.1` and forward `Upgrade` and `Connection` headers to support WebSocket connections.
7. THE NGINX configuration for the `backend_domain` SHALL set `proxy_buffering off` to support server-sent event streaming responses.
8. THE NGINX configuration for the `frontend_domain` SHALL include `proxy_http_version 1.1` and forward `Upgrade` and `Connection` headers to support Next.js WebSocket connections.
9. THE Pipeline SHALL write the generated NGINX configuration to `/etc/nginx/sites-available/cleo` and create a symlink at `/etc/nginx/sites-enabled/cleo`.
10. WHEN the NGINX configuration is written, THE Pipeline SHALL run `nginx -t` to validate the configuration.
11. IF `nginx -t` reports a configuration error, THEN THE Pipeline SHALL output the error and exit with a non-zero status without reloading NGINX.
12. WHEN the NGINX configuration is valid, THE Pipeline SHALL reload NGINX to apply the new configuration.

### Requirement 9: Deploy Action — Docker Compose Deployment

**User Story:** As a DevOps engineer, I want the pipeline to rebuild and restart only the targeted service(s) on re-deploys, so that I can update backend or frontend independently without restarting the entire stack.

#### Acceptance Criteria

1. WHEN `deploy_target` is `all`, THE Pipeline SHALL execute `docker compose -f docker-compose.observability.yml up --build --no-cache -d` to bring up all services.
2. WHEN `deploy_target` is `backend`, THE Pipeline SHALL execute `docker compose -f docker-compose.observability.yml up --build --no-cache -d backend` to rebuild and restart only the backend service.
3. WHEN `deploy_target` is `frontend`, THE Pipeline SHALL execute `docker compose -f docker-compose.observability.yml up --build --no-cache -d frontend` to rebuild and restart only the frontend service.
4. WHEN the `action` input is `deploy`, THE Pipeline SHALL set the `BACKEND_VERSION` environment variable to the value of the `backend_version` input before running `docker compose up`.
5. WHEN the `action` input is `deploy`, THE Pipeline SHALL set the `FRONTEND_VERSION` environment variable to the value of the `frontend_version` input before running `docker compose up`.
6. THE Docker_Compose SHALL use the `BACKEND_VERSION` environment variable to build and tag the backend service image (e.g. `cleo-backend:$BACKEND_VERSION`).
7. THE Docker_Compose SHALL use the `FRONTEND_VERSION` environment variable to build and tag the frontend service image (e.g. `cleo-frontend:$FRONTEND_VERSION`).
8. THE Pipeline SHALL always pass `--no-cache` to the Docker build so that every deploy picks up the latest code and environment changes.
9. THE Docker_Compose SHALL attach all services to the `cleo-network` Docker bridge network.
10. WHEN the deploy completes, THE Pipeline SHALL run health checks only for the services included in `deploy_target` (or all three when `deploy_target` is `all`).
11. WHEN `deploy_target` is `all` or `backend`, THE Pipeline SHALL verify the backend service is healthy by checking the `/health` endpoint on port 8000 with a timeout of 60 seconds.
12. WHEN `deploy_target` is `all` or `frontend`, THE Pipeline SHALL verify the frontend service is reachable on port 3000.
13. WHEN `deploy_target` is `all`, THE Pipeline SHALL verify the Grafana service is reachable on port 3002.
14. IF any targeted service health check fails, THEN THE Pipeline SHALL output the Docker Compose logs for the failing service and exit with a non-zero status.

---

### Requirement 10: Deploy Action — Public IP Output

**User Story:** As a DevOps engineer, I want the pipeline to output the VM's public IP address after a successful deploy, so that I can create DNS A records pointing the configured subdomains to the VM.

#### Acceptance Criteria

1. WHEN the Deploy_Action completes successfully, THE Pipeline SHALL output the `vm_public_ip` value captured from Terraform outputs as a GitHub Actions job output.
2. THE Pipeline SHALL print the VM public IP address in the GitHub Actions job summary in the format: `VM Public IP: <ip_address>`.
3. THE Pipeline SHALL print the configured domain names alongside the public IP in the job summary so DNS records can be created.
4. THE Pipeline SHALL print the deployed `backend_version` and `frontend_version` values in the job summary alongside the VM IP and domain names.

---

### Requirement 11: Destroy Action — Application Teardown

**User Story:** As a DevOps engineer, I want the pipeline to cleanly stop and remove all CLEO services before destroying infrastructure, so that the VM is in a clean state when Terraform destroys it.

#### Acceptance Criteria

1. WHEN the `action` input is `destroy`, THE Pipeline SHALL SSH into the VM using the `VM_SSH_PRIVATE_KEY` GitHub Actions secret and the `vm_public_ip` from Terraform outputs.
2. WHEN the `action` input is `destroy`, THE Pipeline SHALL execute `docker compose -f docker-compose.observability.yml down --volumes --remove-orphans` from the project directory on the VM.
3. WHEN the `action` input is `destroy`, THE Pipeline SHALL remove the NGINX site configuration at `/etc/nginx/sites-available/cleo` and its symlink at `/etc/nginx/sites-enabled/cleo`.
4. WHEN the NGINX configuration is removed, THE Pipeline SHALL reload NGINX to deactivate the removed virtual hosts.
5. WHEN the `action` input is `destroy`, THE Pipeline SHALL remove the `backend/.env` and `frontend/.env.local` files from the VM project directory.
6. IF the NGINX configuration file does not exist during destroy, THEN THE Pipeline SHALL skip the NGINX removal step without failing.
7. IF the env files do not exist during destroy, THEN THE Pipeline SHALL skip the env file removal step without failing.
8. IF the SSH connection to the VM fails during destroy (e.g., VM already gone), THEN THE Pipeline SHALL log a warning and proceed to the Terraform destroy step.

---

### Requirement 12: Destroy Action — Infrastructure Teardown

**User Story:** As a DevOps engineer, I want the pipeline to destroy all Azure infrastructure via Terraform after application teardown, so that cloud resources are fully deprovisioned and no costs are incurred.

#### Acceptance Criteria

1. WHEN application teardown completes during the Destroy_Action, THE Pipeline SHALL run `terraform destroy -auto-approve` in the `terraform/env/dev` working directory with `TF_VAR_postgres_password` set from the `POSTGRES_PASSWORD` GitHub Actions secret.
2. WHEN Terraform destroy completes successfully, THE Pipeline SHALL print a confirmation message in the job summary indicating all resources have been destroyed.
3. IF Terraform destroy fails, THEN THE Pipeline SHALL fail with a descriptive error message listing the resources that could not be destroyed.

---

### Requirement 13: GitHub Secrets Management

**User Story:** As a DevOps engineer, I want all sensitive credentials to be stored as GitHub Actions secrets, so that no secrets are exposed in repository code, pipeline logs, or job summaries.

#### Acceptance Criteria

1. THE Pipeline SHALL read Azure credentials from the `AZURE_CREDENTIALS` GitHub Actions secret for all Azure CLI and Terraform operations.
2. THE Pipeline SHALL read the PostgreSQL password from the `POSTGRES_PASSWORD` GitHub Actions secret and pass it as `TF_VAR_postgres_password` to Terraform.
3. THE Pipeline SHALL read the VM SSH private key from the `VM_SSH_PRIVATE_KEY` GitHub Actions secret for destroy and re-deploy operations.
4. THE Pipeline SHALL read a GitHub Personal Access Token from the `GH_PAT` GitHub Actions secret to authenticate GitHub API calls for writing secrets.
5. WHEN a first-time deploy completes successfully, THE Pipeline SHALL use the `GH_PAT` to call the GitHub REST API (`PUT /repos/{owner}/{repo}/actions/secrets/VM_SSH_PRIVATE_KEY`) to store the Terraform-generated SSH private key, so it is available for all future re-deploys and destroy operations without manual intervention.
6. THE Pipeline SHALL mask the SSH private key value in all pipeline log output.
7. THE Pipeline SHALL never print the contents of any GitHub Actions secret in the job summary or step output.
